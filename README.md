# Урок 1. Подготовка к работе

## Введение
Вам предстоит написать на Python легендарную игру «Тамагочи». Её можно написать по-разному, но ваше решение должно быть реализовано на принципах ООП.

Ваша задача — реализовать часть логики игры, написать необходимые классы и имплементировать методы.

## Описание проекта
Сразу стоит сказать, что у нас будет не совсем обычный Тамагочи, а модифицированный и немного осовремененный (Ubisoft бы позавидовала).
Главное отличие - IDLE механика, которая будет обеспечивать динамику игре.
Суть игры - мы создаем питомца (кто это будет и как будет его имя - это предстоит решать вам), следим за его здоровьем, кормим и укладываем спать.
Но для всех действий требуются деньги (как в жизни, надо же) и деньги будут зарабатываться соответствующим действием - кликом или работой (*тут нужно подумать*).
Таким образом получается полу-тамагочи полу-кликер.


## Правила игры

Давайте поговорим о правилах игры:
- У питомца есть несколько показателей: голод, усталость, здоровье и энергия.
- Питомец может заболеть, это влияет на его здоровье и усталость.
- При падении здоровья ниже 0 игра заканчивается.
- Каждое действие влечёт за собой изменение какого-либо состояния питомца. Например, если мы кормим питомца, то у него уменьшается голод, но может уменьшиться и энергия.

## Общая логика и компоненты игры
Сначала мы будем писать основную логику игры, никак не привязываясь к тому, где и как она будет работать (с красивой графикой, в окне или в консоли и т. д.). Игра будет разделена на несколько компонентов и сущностей: 

- Питомец (тамагочи) — отвечает за логику, связанную с питомцем.
- Еда — содержит данные о конкретной еде и её стоимости, применяется для кормления питомца.
- Лекарство — содержит данные о лекарстве и его стоимости, применяется для лечения питомца.
- Кликер — логика игры, отвечающая за получение монеток, которые можно потратить на еду и лекарства.
- Сущность игры — содержит логику по запуску игры, следит за её состоянием и при наличии соответствующих условий завершает её.

Отдельно стоит обсудить взаимодействие этих компонентов. Игра при этом не знает, что происходит при выполнении того или иного действия, но может знать, какие параметры передаются и что ожидается на выходе функций. За саму реализацию уже отвечают конкретные компоненты, они управляют сущностями питомца и осуществляют менеджмент ресурсов (лекарство, еда, деньги). 

## Файловая структура

Для начала нужно форкнуть данный репозиторий и склонировать его себе. 

Подробнее разберём, что и где лежит.
Проект имеет следующую структуру:
```
├── game
│   ├── clicker.py  <--------- тут пишем логику кликера
│   ├── exceptions.py  <----- тут будут исключения для игры
│   ├── game.py  <---------- тут будет логика игры
│   ├── models.py  <---------- здесь описываем модели игры (еда, лекарство)
│   └── tamagochi.py  <------- логика тамагочи
├── .gitignore
├── main.py  <---------------- точка входа в игру
```

В файле `clicker.py` хранится интерфейс нашего кликера, и там же вы будете писать реализацию для него.

В файле `exceptions.py` описываются кастомные ошибки, чтобы их было проще ловить и обрабатывать более понятным образом.

Файл `game.py` содержит интерфейс для объекта игры, и там же будет писаться реализация для него.

Файл `models.py` описывает модели для нашей игры (еда, лекарства). Они уже написаны, ими мы будем пользоваться в других компонентах.

Внутри файла `tamagochi.py` описан интерфейс для логики питомца, тут же вы будете её реализовывать в классе-наследнике.

В качестве точки входа будет использоваться файл `main.py` с прекодом в виде событийного цикла игры. Он представляет собой бесконечный цикл, в котором на каждой итерации будет ожидаться действие от пользователя. В зависимости от выбранного действия будет отрабатывать соответствующая логика.

Теперь посмотрим на вспомогательные команды, которые вы будете использовать для разработки.

Для проверки кода с точки зрения стилистических ошибок будем использовать линтер `ruff`. Он запускается следующей командой:
```bash
ruff check ./game
```


Если линтер обнаружит ошибки, исправьте код и вновь выполните проверку.

## Классы, методы, функции, игровой цикл

Весь код проекта разбит на классы.
Для основных сущностей определены интерфейсы, которые вы должны будете реализовать в своей игре.
Давайте разберем основные сущности и интерфейсы:

1) `AbstractClicker` - интерфейс для кликера. Методы:
  - `def click(self) -> None:` - основная логика кликера
  - `def __init__(self, income_per_click: int) -> None:` - инициализация кликера

2) `AbstractGame`- интерфейс для сущности игры. Методы:
  - `def __init__(self, tamagochi: AbstractTamagochi, clicker: AbstractClicker):` - инициализация игры
  - `def click(self):` - логика действия "клик"
  - `def buy_food(self, food: Food, price: int):` - логика действия "покупка еды"
  - `def buy_medicine(self, medicine: Medicine, price: int):` - логика действия "купить лекарство"
  - `def feed_tamagochi(self, index: int):` - логика действия "покормить питомца"
  - `def heal_tamagochi(self, index: int):` - логика действия "вылечить питомца"
  - `def rest_tamagochi(self):` - логика действия "отдых"
  - `def play_with_tamagochi(self):` - логика действия "поиграть с питомцем"
  - `def get_status(self):` - получение статуса игры
  - `def food(self) -> list[Food]:` - свойство для доступа к еде
  - `def medicine(self) -> list[Medicine]:` - свойство для доступа к лекарствам

3) `Food` - класс-модель для представления еды. Имеет следующие поля-аттрибуты:
  - `name: str` - наименование еды
  - `satiety: int` - сколько насыщения дает еда

4) `Medicine` - класс-модель для представления лекарства. Имеет следующие поля-аттрибуты:
  - `name: str` - наименование лекарства
  - `heal_hp: int` - сколько лечит здоровья
  - `number_of_uses: int` - максимальное количество использований лекарства
  - `uses: int = 0`  - текущее количество использований. Не может быть больше `number_of_uses`
  Методы:
  - `def is_empty(self) -> bool:` - логика проверки остались ли еще использования лекарства

5) `AbstractTamagochi` - интерфейс для тамагочи. Методы:
  - `def feed(self, food: Food) -> None:` - накормить питомца
  - `def play(self) -> None:` - поиграть с питомцем
  - `def rest(self) -> None:` - отдохнуть
  - `def heal(self, medicine: Medicine) -> None:` - вылечить питомца
  - `def status(self) -> dict[str, int]:` - получить данные о всех показателях питомца
  - `def is_alive(self) -> bool:` - метод для проверки жив ли питомец
  - `def is_sick(self) -> bool:` - метод для проверки болен ли питомец
  - `def update(self) -> None:` - метод для обновления состояния питомца. Должен вызываться каждый тик (или каждое действие)

## Команды для работы с проектом

В шаблон проекта уже встроены различные инструменты - фреймворк для тестов `pytest` и линтеры `ruff` и `flake8`. Далее подробнее рассмотрим как подготовить себе рабочее виртуальное окружение и установить эти инструменты.

Для начала, нужно создать виртуальное окружение. Это можно сделать через стандартный модуль Python следующей командой:
```bash
python -m venv .venv
```
В результате выполнения будет создана директория `.venv` с виртуальным окружением для проекта. 

Чтобы установить необходимые зависимости (pytest, ruff и т.п) нам сначала нужно активировать виртуальное окружение, а затем начать установку из файла `requirements.txt`.

Для активации виртуального окружения, в зависимости от операционной системы, используются команды:
- Linux/MacOS:
```bash
source ./.venv/bin/activate  
```
- Windows:
```ps
.\.venv\Scripts\activate.bat
```


После выполнения активации приглашение командной строки изменится и будет начинаться с названия виртуального окружения: `(.venv) → `

И, наконец, чтобы установить зависимости используем модуль `pip`, встроенный в Python:
```
pip install -r requirements.txt
```

Проверить список установленный в виртуальное окружение пакетов можно командой `pip list`.

Для запуска тестов используем следующую команду:
```bash
pytest ./tests
```

Запуск `flake8`:
```bash
flake8 ./game
```

Запуск `ruff`:
```bash
ruff check ./game
```
